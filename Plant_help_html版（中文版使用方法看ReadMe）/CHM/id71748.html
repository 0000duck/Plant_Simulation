The Waituntil-Statement and the Stopuntil-Statement
SimTalk Reference > General SimTalk Functions > Control Flow Statements > Suspending Methods > The Waituntil-Statement and the Stopuntil-Statement > The Waituntil-Statement and the Stopuntil-Statement

Syntax:

 
waituntil condition [prio number] [wait timespan:time]
stopuntil condition [prio number] [wait timespan:time]

The waituntil/stopuntil-statement suspend method execution until the condition set in the statement evaluates to true. This means that the simulation continues running and that other methods can be executed during this time. As soon as the condition is met, the suspended method will be woken up immediately and its execution will be continued. If methods are being executed at the point in time at which the condition is met, their execution will be interrupted and will only then be continued after the method, that has been woken up, has been executed all the way.

The statements consists of:

  • The keyword waituntil or stopuntil

  • A condition (a boolean expression)

  • The optional keyword prio, and an integer expression used to analyze the priority.

  • The keyword wait if you want to set a time limit after which the statement will be woken up although the condition has not been met yet. You can use the keyword waitExpired to query if the timespan has elapsed.

  • Do not use these statements within a formula or a DDE-call. There the interpreter cancels the execution of the Method with an error message.

When several suspended methods are to be woken up simultaneously, it might happen, that after the Method with the highest priority has been executed, the condition for the other Methods takes on the value false again.

  • For methods that were suspended with waituntil a new evaluation of the condition and the priorities for each additional Method to be woken up takes place after processing each Method that has been woken up.

  • For methods that were suspended with stopuntil these will be woken up in any case when the condition has been met once. This means that no new evaluation of the condition takes place.

Plant Simulation always evaluates logical connections from left to right. The evaluation terminates as soon as the value of the expression are certain.

Example:

 
waituntil Station.Cont /= void AND Station.Cont.Finished

As long as Station.Cont has the value void, Station.Cont.Finished will not be evaluated and thus does not provoke a runtime error.

When you reset your simulation model, Plant Simulation deletes suspensions of methods, which are located in the same Frame as the EventController or in a sub-Frame of this Frame. It also deletes suspensions of methods, whose caller (the anonymous identifier ?) is located in the same Frame as the EventController or in a sub-Frame of this Frame. This means that when you reset an EventController, Plant Simulation no longer deletes suspensions of methods of other simulation models within the model file.

The waituntil/stopuntil-statement are not event-based, so no EventController is required, as compared to the wait-statement, which does require one.

Note that the method deleteSuspendedMethods deletes all suspensions of all methods.
