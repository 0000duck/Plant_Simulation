Condition
SimTalk Reference > General SimTalk Functions > Control Flow Statements > Suspending Methods > The Waituntil-Statement and the Stopuntil-Statement > Condition

The expression that you enter determines under which condition the interpreter continues executing the method. When the condition is true , the interpreter continues the execution with the next statement. When the condition is false, the interpreter suspends the method and monitors the individual components of the expression. If one component changes, the interpreter restarts the method, so that the condition can be analyzed again.

As the interpreter has to automatically break up the expression into components that it can observe:

  • You can use the basic operators (+, -, *, /), comparison operators (, =, =, >=, , /=) and brackets.

  • You cannot use Method calls, access to tables, and built-in methods with parameters.

  • You can, but should not, use built-in methods that produce side effects, such as stack.pop because the condition may have to be analyzed frequently.

Let’s illustrate this with an example: A Transporter is to load an MU and to drive on afterwards. However, we do not know if the Transporter or the MU arrives at the station first. To solve this problem, we can either enter exit controls into the Transporter and the MU, checking if the other one is at the station or not, or we can use a single exit control for the Transporter combined with a waituntil -statement: 

waituntilMU_place.occupied prio 1
MU_place.Cont.move (@)

By using the waituntil -statement, the exit control makes sure that an MU is available to be loaded. If no MU is available, the interpreter suspends the method until an MU arrives, and activates the method as soon as the MU enters the location MU_place.
